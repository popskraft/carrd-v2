# Slider: аудит опций, кода и плавности

Дата: 2026-07-11. Объект: `src/slider/slider.js` (737 строк, min 18.6 KB), `src/slider/slider.css` (190 строк).

## 1. Аудит опций: 17 → 8

Гипотеза подтверждается: из 17 top-level опций реально нужны ~8.

### Оставить

| Опция | Причина |
|---|---|
| `slidesPerView` | Ключевая настройка |
| `gap` | Ключевая настройка |
| `breakpoints` | Ключевая настройка |
| `showDots`, `showArrows` | Часто меняются |
| `autoplay`, `autoplayInterval` | Стандартный кейс |
| `mode` (новая) | `'snap-start' \| 'snap-center' \| 'free'` — см. раздел 3 |

### Убрать / захардкодить

| Опция | Проблема |
|---|---|
| `slideSelector` + `sliderAttribute` | Две опции про одно и то же, никогда не меняются. Захардкодить `data-slider` |
| `peek` | Дублирует дробный `slidesPerView`: в коде это буквально `spv += peek`. Убрать, документировать `slidesPerView: 2.1` |
| `snapThreshold` | Тюнинг-параметр, 0.3 — нормальная константа. Никто не настраивает |
| `maxSlideWidth` | Дублирует контроль ширины через `breakpoints`. Плюс тащит магическое число `736` в двух местах JS + в CSS. Убрать, решать брейкпоинтами |
| `hideOverflow` | Это один CSS-класс. Достаточно CSS-переопределения, из конфига убрать |
| `equalHeight` | Всегда `true` на практике. Захардкодить, отключение — через CSS |
| `freeScroll`, `wheelScroll` | Слить в `mode` (free-режим включает wheel автоматически) |
| `loop` | Сейчас это фейковый loop: не бесшовный, при wrap перематывает анимацией через всю ленту. Либо убрать, либо оставить только для autoplay-перезапуска |

Итог: конфиг сжимается вдвое, инстанс-переопределения (`instances`) остаются как есть — механизм хороший.

## 2. Оценка кода

Общий вердикт: код аккуратный (cleanup, a11y, instance-конфиги — хорошо), но раздут, потому что **вручную реализует то, что браузер умеет нативно**: drag, инерцию, snap, resize-пересчёты. Это ~350 строк из 737.

Конкретные проблемы:

- **Дублирование расчёта ширины**: `updateSlidesPerView()` (CSS calc-строка) и `computeFallbackSlideWidth()` (JS-числа) считают одно и то же двумя путями. Плюс `getSlideWidth()` с тройным fallback.
- **`getRenderedTranslate()`** — ручной парсинг трёх форматов matrix regex'ами. Заменяется одной строкой: `new DOMMatrix(getComputedStyle(track).transform).m41`.
- **Layout thrashing**: на каждый `mousemove` цепочка `clampDragTranslate → getMinTranslate → getSlideWidth → offsetWidth` — принудительный reflow каждый кадр драга. Ширину надо кэшировать на `dragStart`.
- **Дублированные обработчики** touch + mouse вместо Pointer Events (поддержка повсеместная с 2020). Pointer capture заодно чинит обрыв драга при выходе курсора (сейчас `mouseleave` на wrapper жёстко завершает drag).
- **`mousemove` слушается всегда**, не только во время драга (early return спасает, но обработчик с capture висит на каждое движение мыши).
- **Магические числа** разбросаны: `736` (×3 места), `6` (порог оси, ×2), `0.2`, `0.8`, `0.5`, `0.95`, `16`, `450`, `500`.
- Мёртвый вес: `data-slider-id` fallback в `init()` нигде не документирован; `loop` реализован на четверть.

## 3. Режимы поведения: free / snap-center

Подтверждаю наблюдение: **сейчас snap фиксирует слайд по левому краю**, не по центру. `animateTo()`: `translateX = -slideWidth * index` — без учёта центрирования во вьюпорте. `freeScroll` существует, но это отдельный флаг, а не режим.

Предложение — одна опция:

```js
mode: 'snap-start' | 'snap-center' | 'free'
```

Для `snap-center` формула позиции:

```js
offset = (viewportWidth - slideWidth + gap) / 2;
translateX = clamp(-index * slideWidth + offset, min, 0);
```

— слайд встаёт по центру кадра, соседние выглядывают симметрично с двух сторон (тот самый «зазор на следующую иллюстрацию»). На первом/последнем слайде clamp прижимает к краю — либо разрешить «пустые поля» по краям (у Swiper это `centeredSlides + centerInsufficientSlides`), это отдельное решение.

`free` = текущий `freeScroll` + wheel включён + после инерции только синхронизация точек, без доводки.

## 4. Качество плавности — есть реальные причины рывков

1. **Инерция зависит от FPS.** `velocity *= 0.95; translateX += velocity * 16` — константы на кадр, а не на время. На 120 Гц (все современные iPhone/iPad, ProMotion-маки) затухание происходит вдвое быстрее, скорость прироста другая → на разных устройствах слайдер «тормозит» по-разному. Нужен dt-based расчёт через `performance.now()`.
2. **Скорость броска считается по двум последним точкам** через `Date.now()` (низкое разрешение, до 1–2 мс шума). Одно дрожание пальца в последний момент — и fling получает случайную скорость → рывок при отпускании. Правильно: усреднять velocity за последние ~100 мс.
3. **Фиксированная длительность snap 0.4s** на любую дистанцию: доводка на 10px и перелёт на 3 слайда занимают одинаковое время — короткие движения кажутся вязкими, длинные — дёргаными.
4. **Reflow на каждом кадре драга** (п. 2 выше) — прямой источник jank на слабых устройствах.
5. **Разрыв стилей инерции и snap**: инерция — ручной rAF-цикл, доводка — CSS transition. На стыке (finishMomentum → snapToNearest) характер движения меняется скачком.
6. `snapFallbackTimer` 450 мс при transition 400 мс — ок, но это костыль вместо надёжного механизма.

## 5. Главная рекомендация

Рассмотреть перевод на **нативный CSS scroll-snap** (`overflow-x: auto` + `scroll-snap-type: x mandatory` + `scroll-snap-align: start|center`):

- Убирает весь ручной drag/momentum/snap-код (~300–350 строк, минус половина файла).
- Нативная инерция ОС — эталонная плавность, все проблемы п. 4 исчезают как класс.
- `snap-center` получается бесплатно (`scroll-snap-align: center`).
- Free-режим = просто `scroll-snap-type: none`.
- JS остаётся только для: обёртки DOM, точек/стрелок (синхронизация через `scrollend`/IntersectionObserver), autoplay (`scrollTo({behavior:'smooth'})`), брейкпоинтов.

Издержки: скрыть скроллбар (стандартные приёмы), поведение колеса мыши по горизонтали (нативно работает с shift), `scrollend` в Safari — с 2024, есть простой fallback. Для контекста Carrd (маркетинговые страницы, mobile-first) — оптимальный размен.

Если переписывать не хочется — минимальный план: кэш ширины на dragStart, dt-based инерция, усреднение velocity, Pointer Events, режим `snap-center`.

## Резюме

- Опции: 17 → 8, `peek`/`maxSlideWidth`/`freeScroll`/`wheelScroll`/селекторы — убрать или слить в `mode`.
- Код: рабочий, но половина файла — ручная реимплементация нативного скролла; есть дубли и layout thrashing.
- Центровка: сейчас только по левому краю; `snap-center` — простая формула или бесплатно через scroll-snap.
- Рывки: реальны, главные причины — FPS-зависимая инерция и шумный расчёт velocity.
