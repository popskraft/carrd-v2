# Аудит качества — carrd-v2 (плагины и модули)

Дата: 2026-07-07

## Stack Profile

```
Backend:   нет (vanilla JS/CSS плагины) + admincarrd (PHP 8, без фреймворка)
Frontend:  Vanilla JS/CSS, экспорт под Carrd, сборка через Python-скрипты
Admin:     admincarrd — собственный PHP-модуль (auth, CSRF, setup)
Database:  нет (LocalStorage на клиенте в shopping-cart)
Tests:     Node --test (jsdom) — 216 тестов src/, 26 тестов cardbuilder
Build:     python3 scripts/minify_plugins.py → dist/, bundle.config.json (CDN-бандл theme-core)
CSS:       vanilla CSS с design tokens (--theme-<plugin>-*)
CI/CD:     GitHub Actions (.github/) — не проверялось детально
Пакеты:    pnpm (заявлен), но npm audit не работает без npm lock — ожидаемо
```

Репозиторий — это не одно приложение, а три поверхности:
1. **`src/`/`dist/`** — 14 независимых Carrd-плагинов (JS+CSS), общий `theme-config.js`/`theme-ui.css`.
2. **`cardbuilder/`** — Node-автоматизация Carrd (MCP-сервер, CDP-скрипты, ~2600 строк в `lib/`).
3. **`admincarrd/`** — самостоятельный PHP-модуль админки (auth, загрузка, конфиг).

Ниже — оценка по применимым бенчмаркам (Core 1–8, JS/Frontend 14–15, Security Headers 17). PHP/Laravel-бенчмарки (9–13) не применимы — в `admincarrd` нет Laravel/Eloquent.

---

### 1. Code Coverage — 48/100 🟠

**Суть**: доля кода, покрытого тестами, и наличие инструментов измерения.

**Влияние**: без числового покрытия нельзя увидеть регрессии в непокрытых ветках (например, edge-cases валидации в `shopping-cart.js`).

**Текущее состояние**: инструмента покрытия (c8/nyc/istanbul) нет. Зато у **каждого** из 14 плагинов есть отдельный поведенческий тест (`tests-js/<plugin>.test.js`), плюс контрактные тесты (`bundle-contract`, `contracts`, `tokens-consistency`, `idempotency`, `backward-compat`). Итого 216 тестов в `src/`, все зелёные, + 26 в `cardbuilder`. Функциональное покрытие выглядит широким, но без цифр это не проверить.

**Что изменить для 100**:
1. Подключить `c8` к `node --test` (`c8 --reporter=text npm run test:js`).
2. Добавить порог покрытия в CI (старт с 50%, цель 70%+).
3. Точечно добить edge-cases в `shopping-cart.js` (валидация, `validateCartItem` с complexity 12) и `slider.js` (drag-логика).

### 2. Duplication — 85/100 🟢

**Суть**: повторяющийся код между плагинами.

**Влияние**: низкое — изменения не нужно синхронизировать в нескольких местах.

**Текущее состояние**: прогнан `jscpd` (min-lines 5, min-tokens 50) по `src/`. Итог: 3 клона, 0.55% дублированных строк, 0.50% токенов на 7766 строк / 55 файлов. Оба JS-клона и CSS-клон — это совпадения с общим `theme-config.js`/`theme-ui.css`, то есть намеренное переиспользование темы, а не copy-paste багов.

**Что изменить для 100**: не критично; при желании — вынести повторяющийся 16-строчный блок `grid-cluster.js` в `theme-config.js` явным импортом вместо дублирования.

### 3. Coupling — 65/100 🟡

**Суть**: зависимость модулей друг от друга.

**Влияние**: каждый плагин — самостоятельный IIFE/модуль, подключаемый независимо в Carrd; низкая связанность snizhaet риск "one plugin breaks another".

**Текущее состояние**: плагины в `src/` не импортируют друг друга напрямую — связь только через общий `theme-config.js` (конфиг) и CSS custom properties с namespace `--theme-<plugin>-*` (это подтверждено тестом `contracts.test.js`). В `cardbuilder/scripts/carrd/lib/` — разделение на `cdp-client.mjs`, `control-core.mjs`, `onboarding-core.mjs`, `readiness-core.mjs`, `target-map.mjs`, `keygen.mjs` — по функциональным границам. Явных интерфейсов/контрактов (TS types, JSDoc `@typedef`) нет — это vanilla JS, так что формальных контрактов не ожидается, но тестовый файл `contracts.test.js` частично играет эту роль.

**Что изменить для 100**:
1. Задокументировать публичный API каждого плагина через JSDoc `@typedef` (сейчас нет).
2. `onboarding-core.mjs` (802 строки) стоит разбить на более мелкие модули по стадиям онбординга.

---

### 4. Cohesion — 60/100 🟡

**Суть**: у каждого модуля — одна ответственность.

**Влияние**: большие файлы (`shopping-cart.js` 1111 строк, `slider.js` 841, `onboarding-core.mjs` 802) сложнее читать и тестировать локально.

**Текущее состояние**: большинство плагинов компактны (250–450 строк) и решают одну задачу. Исключения: `shopping-cart.js` (1111 строк — валидация + рендер + LocalStorage + форма заказа в одном файле) и `slider.js` (841 строка — drag, autoplay, resize, a11y в одном классе). В `cardbuilder`, `onboarding-core.mjs` также смешивает несколько стадий процесса.

**Что изменить для 100**:
1. В `shopping-cart.js` вынести валидацию (`validateCartItem`) и работу с LocalStorage в отдельные модули/файлы.
2. В `slider.js` выделить drag-логику (`onDragMove`, `getSlideWidth`) в отдельный helper.

---

### 5. Cyclomatic Complexity — 72/100 🟡

**Суть**: количество независимых путей выполнения в функции.

**Влияние**: ESLint уже настроен с `max-complexity: 10` — сигнал зрелой инженерной культуры.

**Текущее состояние**: `npm run lint` даёт **0 ошибок, 12 предупреждений** (complexity 11–20) в 6 файлах: `cards.js` (20), `slider.js` (14, 14, 12), `shopping-cart.js` (12, 11, 11), `modal.js` (17, 11, 11), `faq.js` (13), `stacker.js` (11). Ни одна функция не превышает 20 — критического порога нет.

**Что изменить для 100**:
1. Разбить `cards.js:216` (complexity 20) на подфункции — самый горячий файл.
2. Аналогично для `modal.js:385` (17) и `slider.js:252/321` (по 14).
3. Добавить `check:complexity` как обязательный шаг CI (уже фактически есть через `lint`, но warnings не блокируют — рассмотреть `--max-warnings 0` для новых файлов).

### 6. Dead Code — 55/100 🟠

**Суть**: неиспользуемый код.

**Влияние**: не проверено инструментом статического анализа (knip/depcheck не запускались) — оценка ниже из-за отсутствия инструментальной верификации, а не из-за найденных проблем.

**Текущее состояние**: ручной осмотр не выявил явно мёртвых файлов или закомментированных блоков кода в `src/`. `bundle.config.json` явно исключает `no-loadwaiting/script.js` из сборки — стоит проверить, используется ли этот файл вообще.

**Что изменить для 100**:
1. Прогнать `npx knip` по `src/` и `cardbuilder/scripts` — найдёт неиспользуемые экспорты/файлы.
2. Проверить назначение `src/no-loadwaiting/script.js`, исключённого из бандла.

### 7. Error Handling — 65/100 🟡

**Суть**: последовательность обработки ошибок и отсутствие "тихих" сбоев.

**Влияние**: для клиентских виджетов (без сервера) отсутствие централизованного мониторинга — норма, но лишь если ошибки логируются в консоль, а не глотаются молча.

**Текущее состояние**: `try/catch` встречается в 5 из 14 плагинов (`accordeon`, `cookie-banner`, `no-loadwaiting`, `shopping-cart`, `stacker`) — пустых catch-блоков не найдено, везде есть `console.warn`/`console.error` с контекстом. В `admincarrd` (PHP) `try/catch` присутствует в критичных `Auth.php`, `ConfigWriter.php`, `Optimizer.php`.

**Что изменить для 100**:
1. Добавить `try/catch` вокруг DOM-операций в плагинах без него (`faq`, `modal`, `slider`, `switcher`, `header-nav` и др.) — сейчас ошибка в разметке страницы может уронить весь скрипт.
2. В `admincarrd` — обернуть операции с файловой системой (`ConfigWriter`, загрузка zip) более гранулярными типизированными исключениями.

### 8. Documentation Coverage — 35/100 🟠

**Суть**: доля публичных функций с docblocks/типами.

**Влияние**: у каждого плагина есть `README.md` (пользовательская документация хорошая), но внутри кода JSDoc почти отсутствует — усложняет онбординг нового разработчика в конкретный файл.

**Текущее состояние**: JSDoc (`/**`) найден только в `modal.js` (12 блоков), `shopping-cart.js` (6), `typography.js` (3). Остальные 11 из 14 плагинов — 0 docblocks на публичных методах.

**Что изменить для 100**:
1. Добавить JSDoc к публичному API каждого плагина (обычно 1 `init`/`open`/`destroy` на файл — минимальные трудозатраты).
2. Начать с самых сложных файлов из п.5 (`cards.js`, `slider.js`) — там документация окупится быстрее всего.

---

### 14. Frontend Bundle Health — 60/100 🟡

**Суть**: размер и оптимизация JS/CSS бандлов.

**Влияние**: плагины загружаются в Carrd через CDN (jsDelivr) — вес каждого файла напрямую влияет на скорость сайтов на этой платформе.

**Текущее состояние**: есть управляемый пайплайн сборки (`scripts/minify_plugins.py` → `dist/`), `bundle.config.json` описывает общий CDN-бандл `theme-core` (11 из 14 плагинов) с shared CSS/JS, что снижает дублирование запросов. Анализ бандла (webpack-bundle-analyzer/vite-visualizer) не проводился — по объективным причинам: сборка на Python, не Vite/Webpack.

**Что изменить для 100**:
1. Добавить простой скрипт-отчёт размеров `dist/*.min.js` в CI (порог по КБ на плагин).
2. Проверить, не растут ли `shopping-cart.min.js`/`slider.min.js` (самые крупные исходники) сверх разумного веса после минификации.

### 15. Accessibility (a11y) — 70/100 🟡

**Суть**: соответствие WCAG — семантика, ARIA, клавиатура.

**Влияние**: виджеты (модалки, аккордеоны, слайдеры) часто становятся точкой провала доступности сайтов.

**Текущее состояние**: 10 из 14 плагинов используют ARIA-атрибуты (`aria-expanded`, `aria-hidden`, `aria-label`, `aria-controls`, `aria-pressed`, `aria-labelledby`) — `shopping-cart.js` и `faq.js` лидируют по плотности использования. Последний коммит в git-истории (`e60b160 fix(audit): ... plugin a11y ...`) подтверждает, что доступность уже целенаправленно улучшалась. Автоматизированных a11y-тестов (axe-core) в `tests-js` не найдено.

**Что изменить для 100**:
1. Добавить `axe-core` в jsdom-тесты хотя бы для `modal`, `faq`, `accordeon`, `switcher` (интерактивные виджеты).
2. Проверить `cards`, `grid-cluster`, `stacker`, `typography` — в них ARIA не встречается вообще; уточнить, нужна ли она по семантике компонента.

---

### 17. Security Headers — 75/100 🟡

**Суть**: HTTP security headers.

**Влияние**: применимо к `admincarrd` — единственной серверной поверхности с формами логина/загрузки.

**Текущее состояние**: `admincarrd/app/lib/bootstrap.php` явно ставит `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: no-referrer`, `Content-Security-Policy` (кастомная политика). Отсутствует `Strict-Transport-Security` (HSTS). `.htaccess` дополнительно блокирует прямой доступ к `app/`, `var/`, `docs/`, `config.php`, `router.php` — хорошая практика defense-in-depth.

**Что изменить для 100**:
1. Добавить `header('Strict-Transport-Security: max-age=31536000; includeSubDomains')` в `bootstrap.php` (только если сайт всегда на HTTPS).
2. Прогнать реальный домен через securityheaders.com после деплоя для подтверждения grade A.

## Scope

Проанализировано: `src/` (14 плагинов), `tests-js/`, `cardbuilder/scripts/carrd/` (включая `lib/`), `admincarrd/app/` (auth, csrf, config).

Не анализировалось в этом прогоне: `node_modules/` (сторонний код — вне области), `dist/` (генерируемый артефакт — оценивается через `src/`), `carrd-source/` (референсный шаблон, не собственный код), содержимое `cardbuilder/projects/*/data` (данные конкретных сайтов, не код), `.github/` workflows (CI-конфигурация не разбиралась построчно).
