# Практическая задача: перенос контента Site 2 -> Site 1

## Контекст

- Site 1 (target, новый дизайн-оригинал): `https://carrd.co/dashboard/8089177104819774/build`
- Site 2 (source, устаревшая версия): `https://carrd.co/dashboard/4544177104830762/build`

Цель: перенести контентные элементы из Site 2 в Site 1 в зону между Header и Footer (сразу после Header), сохранив управляемость и возможность быстро применить стили Site 1.

## Что должно получиться

1. Полный снимок структуры и стилей Site 1.
2. Полный снимок структуры Site 2.
3. Карта соответствия элементов:
   - `site2:data-id -> site1:data-id` (или отметка "new element required").
4. Перенесенные элементы из Site 2 в Site 1 в целевую область (после Header End).
5. Отчет о том, что перенесено автоматически, а что оставлено дизайнеру.

## Базовые правила

- Использовать стабильные маркеры: `data-id`, `data-type`, `data-action`.
- Не опираться на индексы кнопок/вкладок.
- Делать перенос батчами с откатом через Undo boundaries.
- Не выполнять Save/Publish автоматически. Сохранение и публикация только оператором.

## Технический пайплайн

### Этап 1. Сбор данных

1. На Site 1 выполнить `carrd-deep-audit-style-map.js`.
2. На Site 2 выполнить `carrd-deep-audit-style-map.js`.
3. Сохранить два JSON-файла в `data/raw-imports/`.

Ожидаемые артефакты:
- `site1-deep-snapshot-*.json`
- `site2-deep-snapshot-*.json`

### Этап 2. Нормализация и карта

1. Построить таблицу типов и порядка верхнеуровневых блоков.
2. Выделить границы секций по control-элементам:
   - `Header End`
   - `Footer Start`
3. Определить целевую точку вставки: первый контентный блок после `Header End` в Site 1.
4. Сформировать mapping:
   - exact match (по типу + текстовому маркеру),
   - partial match,
   - no match (нужно создать элемент).

### Этап 3. Перенос контента

1. Скопировать/создать отсутствующие элементы из Site 2 в Site 1.
2. Вставлять элементы сразу после Header End в целевой области.
3. Сохранять журнал операций:
   - `source data-id`,
   - `target data-id`,
   - тип операции (create/update/move/skip),
   - статус.

### Этап 4. Применение стилей Site 1

Вариант A (предпочтительно): назначить существующие style tokens Site 1 новым/перенесенным элементам.  
Вариант B (допустимо): остановиться после переноса контента и передать задачу дизайнеру для ручной стилизации.

Правило для несовместимых стилей:
- если стиль из Site 2 отсутствует в Site 1, назначать дефолтный стиль контейнера `Main`.

### Этап 5. Валидация

Проверки:
- структура после Header не ломает секции;
- контент виден в Desktop и Mobile view;
- критичные элементы не скрыты visibility-настройками;
- нет случайного удаления существующих блоков Site 1.

## Минимальные deliverables

1. `MIGRATION_MAPPING.csv` или `MIGRATION_MAPPING.md`.
2. `MIGRATION_LOG.md` (пошаговые действия и статус).
3. `VALIDATION_CHECKLIST.md`.

## Критерии "Done"

- Перенесены согласованные блоки из Site 2 в Site 1 в нужную зону.
- Для каждого перенесенного блока есть запись в mapping/log.
- Есть явный список блоков, отданных на ручной дизайн.
- Страница Site 1 остается рабочей и структурно валидной.

## Утвержденные решения

1. Переносить весь контент из Site 2 между Header и Footer.
2. Форму (`form`) не переносить.
3. Секции `#order` и `#thankyou` не переносить.
4. Для стилей, отсутствующих в Site 1, назначать дефолтный стиль контейнера `Main`.
5. Save/Publish агент не выполняет, это делает только оператор.
