# Открытые Вопросы

## Суть
`slider-v2` принят владельцем и промотирован в единственный `slider` (Q012–Q014 закрыты, см. «Решённые»). Локальные релизные gates миграции JS-опций на data-* пройдены, но live launch path всё ещё блокируется на границе Carrd Builder -> publish -> buyer install/support. С `2.0.0` (Q016 закрыт) доставка — только inline embed: CDN/jsDelivr и legacy compat-слой удалены полностью, нет больше временных compatibility tails.

Стабильная точка продолжения `2026-07-13`: canonical draft `main-template` после MBX reinstall/remap и cleanup probe-атрибутов считается текущей безопасной baseline для следующей сессии. Возобновлять работу нужно с remaining persistence/apply matrix, а не с повторной пересборки canonical draft.

## Ядро
- Q015 [open] Миграция JS-опций на data-* (`docs/specs/plugin-options-data-migration-plan.md`, ROADMAP #62) локально завершена, а live persistence path уже частично доказан `2026-07-13`: harmless custom `data-*` на stable `container18` пережил `Save -> Reload -> Readback`, cleanup пережил `Publish -> Reload -> Published Readback`; safe custom `id` на `container28` пережил `Publish -> Builder Readback -> Published HTML`, cleanup тоже пережил этот путь. Но end-to-end proof для реальных product attrs не закрыт и уже есть первый negative signal: `data-modal-close-on-overlay=off` на реальном modal target `container12` сохранился в Builder после publish, однако опубликованный `#data-modal-contact` этот attr не получил. Нужно отдельно подтвердить, это modal-specific filtering или общий publish/runtime barrier для `data-shopping-cart-checkout-target`, `data-modal-close-on-overlay`, `data-slider-arrows-mobile`.
- Q027 [open] Как именно продолжать verification от стабильной точки `2026-07-13`, не теряя уже доказанную baseline? Следующий цикл должен идти по фиксированной очереди: `data-slider-arrows-mobile` на реальном target, затем `data-shopping-cart-checkout-target`, затем controlled embed edit, затем structural/apply proof. Повторный full remap canonical draft не нужен, если владелец снова не меняет Builder structure вручную.
- Q017 [open] Упростить buyer install flow так, чтобы пользователь не ошибался с placement embeds в Carrd. Нужен один канонический buyer-facing install runbook, где `Head`/`Body End` placement, split embeds и visibility rules проверяются пошагово на общем макете.
- Q018 [open] Снизить риск неправильного использования clean `data-*` contract покупателем. Нужны buyer-facing examples, compatibility matrix и явные правила, какие `data-*` обязательны, какие optional, где их ставить нельзя, и как быстро проверить, что Carrd их реально сохранил.
- Q019 [open] Зафиксировать buyer-facing scope продажи: что именно входит в шаблон/пакет, что не входит, кто делает install/publish/support, и нужен ли managed rollout path вместо self-service. Без этого покупателю непонятно, что он реально покупает.
- Q020 [open] Оформить formal troubleshooting/support flow для покупателя: какие данные он должен прислать (`URL`, browser, screenshot, console/network errors, embed inventory, current version), что считается support boundary, и как выглядит rollback path.
- Q021 [open] Явно ограничить product promise для `shopping-cart`: это client-side order UI/form serializer, а не полноценный ecommerce backend/payment system. Нужна единая формулировка в sales/install/support docs, чтобы покупатель не ожидал checkout/payment/database возможностей, которых в repo нет.
- Q022 [open] Закрыть published release drift на canonical smoke surface. Исторический direct `quickPublish()` failure от `2026-07-11` уже не является абсолютным blocker: `2026-07-13` cleanup publish на canonical Builder прошёл и post-reload/readback был чистым. Но published drift всё ещё не закрыт содержательно: нужно перевести published `mini.crd.co` на актуальный contract, добиться зелёного `npm run check:published` и повторить full live verify.
- Q023 [open] Доказать полный published browser smoke для canonical sales surface. Локальные `npm run validate` и local browser-lab зелёные, но published `mini.crd.co` всё ещё не покрыт полной матрицей critical user journeys после текущего launch-blocking drift.
- Q024 [open] Решить, нужен ли отдельный formal quality gate для `admincarrd/**` на уровне coverage/static/security. Основной runtime (`src/**`, `cardbuilder/**`) уже защищён намного сильнее, чем `admincarrd`, и это остаётся launch-risk для tooling side.
- Q025 [open] Убрать portability drift в active docs: часть `cardbuilder/docs/**` и связанных owner-docs всё ещё содержит абсолютные локальные пути `/Users/popskraft/Projects/carrd-v2/...`, что ухудшает переносимость knowledge package и buyer/operator handoff.
- Q026 [open] Владелец решил (2026-07-11): доступ к старым/будущим CDN-версиям плагинов через jsDelivr будет жить в отдельной ветке и не будет публичным — тема закрыта для текущей `2.0.0` embed-only линии, действие отложено. Когда/если CDN-доставка вернётся, это `3.0.0` (см. `docs/specs/release-contract.md`) и потребует отдельного контракта для этой ветки (кто имеет доступ, как публикуется, отношение к публичному `main`).

## Детали
Решённые архитектурные вопросы оформляются как ADR в `docs/decisions/` и удаляются из этой таблицы. Вопросы по tooling вне runtime repo ведутся в workspace, которому принадлежит этот tooling.

## Решённые
- Q016 → владелец принял `inline embeds by default`, но в более радикальной форме, чем изначально сформулировано: CDN/jsDelivr не остался advanced path, а удалён полностью (`2.0.0`, коммит `feat!: switch to inline-embed-only distribution`). `README.md`, `AGENTS.md`, `docs/specs/release-contract.md`, `docs/specs/carrd-contract.md` и все plugin README обновлены под единственный Inline Embed путь.
- Q012 → владелец принял механику `slider-v2` (scroll-snap, drag, autoplay) 2026-07-11: split на 2 embed-части оставлен как есть (`terser` даёт ~20% сокращение, недостаточно чтобы убрать сплит); `mode=start` отложен без срока; edge-clamp без inline-padding пояснён в чате и принят владельцем как есть 2026-07-11 (первый/последний слайд прижаты к краю, между слайдами зазор с обеих сторон — следствие `scroll-snap-align: center`, упирающегося в границы скролл-контейнера); добавлена `data-slider-arrows-mobile="on"` (включает стрелки на мобильных, где раньше они были безусловно скрыты). Владелец распорядился (2026-07-11): «сделай вторую версию основной, а первую — в архив». Выполнено на уровне `src/`: содержимое `src/slider-v2/` (адаптированное — комментарии/warn-префикс `[slider-v2]` → `[slider]`) записано в `src/slider/{slider.js,slider.css,README.md,demo.html}`; оригинальный v1 (`slider.js`/`slider.css`/`README.md`) сохранён без изменений в `docs/_archive/slider-v1-source/`; `tests-js/slider.test.js` заменён на адаптированную версию `tests-js/slider2.test.js` (пути/описания переименованы). `scripts/minify_plugins.py`: `SPLIT_EMBED_PLUGINS` больше не содержит `"slider-v2"` (только `"slider"`). Единая конвенция data-* (пробельные триплеты vs суффиксы `-sm`/`-lg`) осталась нерешённой как общий вопрос стиля на будущее, не блокирует промоушен. Ограничение среды: папка `src/slider-v2/` и файл `tests-js/slider2.test.js` физически остались на диске как дубликаты (sandbox не позволяет удалять/переименовывать файлы в этой сессии) — требуется либо ручная очистка владельцем, либо запрос permission через `allow_cowork_file_delete`. Пересборка `dist/` не выполнялась в этой сессии (см. Q015).
- Q013 → снято как отдельный вопрос: после промоушена (см. Q012) `slider-v2` как отдельная dist-цель больше не актуален — `slider` (бывший v2) остаётся единственным в общем бандле `theme-runtime`, `bundle.config.json`/`cdn_bundle.plugins` не менялись (уже содержали `"slider"`).
- Q014 → снято как отдельный вопрос: унификация namespace v2↔v1 (`window.CarrdSlider`, `.theme-slider-*` классы), сделанная заранее в ожидании промоушена, подтвердила свою полезность — при промоушене (см. Q012) не потребовалось никаких изменений рантайм-имён.
- Q005 → `docs/decisions/canonical-live-smoke-surface.md` (`main-template` Builder + `mini.crd.co` published являются единым canonical smoke surface; `faktura` остаётся secondary integration surface, `faktura-app` не считается Carrd smoke surface).
- Q002 → `docs/decisions/switcher-target-only-contract.md` (cluster mode удаляется полностью; сначала `main-template` переводится на `data-switcher-target` + explicit indexes, затем удаляются runtime/config/tests/docs).
- Q003 → `docs/decisions/shopping-cart-checkout-contract.md` (`data-shopping-cart-target` удалён как неиспользуемый marker; checkout contract — Section Break `shopping-cart`, form `form-shopping-cart`, textarea `data-shopping-cart-output="order-details"`).
- Q004 → `docs/decisions/clean-runtime-contract.md` (владелец выбрал clean source/runtime contract для `carrd-v2`; compatibility tails разрешены только как repo-owned rollout artifacts `theme-core.*` / `theme-ui.css` до завершения controlled migration, старая линия остаётся в `popskraft/carrd-plugins`).
- Q008 → `docs/decisions/legacy-alias-feature.md` (legacy `[data-*]` alias-фича завершена, закоммичена, опубликована и live-verified).
- Q009 → `test:cardbuilder` переведён в machine-independent режим: `tests-js/site-registry.test.js` больше не шьёт абсолютные repo paths, core entrypoints `cardbuilder/scripts/carrd/open-debug-chrome.sh` и `refresh-builder-plugins.mjs` вычисляют repo root динамически, а `test:cardbuilder` включён в default `npm run test`.
- Q011 → `tests-js/accordeon.test.js` снова зелёный (11/11); прежний fixture-дрейф carrd-source устранён. `tests/test_admincarrd_*` php-тесты теперь `skipUnless(shutil.which("php"))`, поэтому `npm run test` машинно-независим и без php (skipped, не error).
- Q010 → канонический атрибут `data-stacker` подтверждён как единственная активная форма; legacy alias `data-stacked` в runtime сохранён для внешних/устаревших разметок, но больше не используется в active markup. `main-template` Builder мигрирован (`container22`/`container23`/`container24`, группа `stack`): `data-stacked=stack` → `data-stacker=stack`, опубликовано и live-verified на `mini.crd.co` 2026-07-07.
- Q004-follow-up → theme token contract split (`theme-runtime` + `theme-design-tokens-embed.html`) опубликован на `main-template`/`mini.crd.co` 2026-07-07 (тег `v2.1.0`): `embed14`/`embed02` переведены с `theme-core@main` на `theme-runtime@2.1.0`, новый Head-embed `embed03` несёт токены-примитивы; live-verified — все `--theme-*` токены резолвятся, 6 slider / 3 switcher / 3 stacker / 1 modal без регрессий.
- Q004-addons → add-on плагины приведены к контракту «один embed на плагин, CDN, version-pinned» и опубликованы на `mini.crd.co` 2026-07-07: свалка `embed01` (инлайновый shopping-cart JS + cookie-banner, `@main`) удалена и заменена на `embed04` (Shopping Cart CDN) + `embed05` (Cookie Banner CDN), `no-loadwaiting` (`embed08`) допинан до тега. Live-verified: **ноль** `@main`-ссылок, все 7 CDN-ресурсов на `@2.1.0`, без инлайн-дублей и двойной инициализации.
- Q013 → `slider-v2` подключён в `dist/` как individual (не bundled) плагин, параллельно с v1 `slider`: временный `DIST_BUILD_EXCLUDES += "slider-v2"` (использовавшийся во время prototype-only фазы) снят, `dist/slider-v2/*` собран обычным `npm run build`, `bundle.config.json`/`cdn_bundle.plugins` не тронуты (v1 `slider` остаётся единственным в общем бандле `theme-runtime`). `tests/test_minify_plugins.py` (`test_repo_generated_readmes_match_bundle_contract`) по-прежнему пропускает слаги из `DIST_BUILD_EXCLUDES` — механизм оставлен как есть для будущих подобных случаев (как `columns`), сейчас пуст для `slider-v2`.
- Q014 → `slider-v2` runtime-namespace унифицирован с v1: `CarrdSlider2` → `window.CarrdSlider`, `.theme-slider2*` классы → `.theme-slider-wrapper`/`-scroller`/`-slide`/`-dots`/`-dot`/`-nav`/`-nav--prev`/`-nav--next`, класс `Slider2` → `Slider`, CSS-переменные `--s2-gap`/`--s2-slide-w` → `--slider-gap`/`--slider-slide-w`. `data-slider`/`data-slider-initialized` и так были общими с v1 (перенесены как есть). Причина: владелец подтвердил, что v1 и v2 никогда не будут установлены на одной странице одновременно (полная замена на живом сайте, не сосуществование), поэтому нет смысла держать временный "2"-суффикс, который потом всё равно пришлось бы убирать. `slider-v2` как имя папки/dist-слага/заголовка README не менялось — унификация только внутри рантайм-кода. Оба README прямо документируют несовместимость при одновременной установке. `docs/specs/slider-v2-plan.md` содержит addendum с датой, фиксирующий это отклонение от изначально описанных в спеке имён.
